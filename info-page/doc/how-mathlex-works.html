How MathLex Works

MathLex works in two phases. The first phase compiles a MathLex expression into an Abstract Syntax Tree (AST) that can be represented in memory, and the second phase converts the AST into some type of output.

MathLex → Syntax Tree

When provided with a valid MathLex string, MathParser.parse() produces an abstract syntax tree (AST) representing the inferred value of the MathLex code.

The AST is built from different "node" types represented as a recursive array. Every node has at least one element (i.e. the 0th element or node[0] in JS) that is a string indicator of the type of node. Appendix B lists all possible nodes and their structure.

Under the hood, MathLex has two components: a Tokenizer and a Parser. The Tokenizer is responsible for translating the characters in the MathLex input string into a list of Tokens, a way to group related characters into a single symbol. For example, "<=" is shorthand for "less than or equal to" (in display math, '≤') and is comprised of two separate characters. The Tokenizer groups these characters into a TLessEqual for the parser.

The Parser then reads the list of tokens and assembles the corresponding abstract syntax tree (AST) based on the rules in src/grammar.coffee. The parser code itself is generated by Jison and for most purposes is unreadable.
